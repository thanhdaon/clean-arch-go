## Project Overview

This is a **Task Management API** built with **Clean Architecture** principles in Go. It implements role-based access control for Employers and Employees, with comprehensive task assignment and management capabilities. The project follows strict Clean Architecture with hexagonal ports & adapters pattern.

## Development Commands

### Essential Setup
```bash
# Install required tools
go install github.com/oapi-codegen/oapi-codegen/v2/cmd/oapi-codegen@latest
go install github.com/rubenv/sql-migrate/...@latest

# Start MySQL database (required)
docker compose up -d  # or manually start MySQL on localhost:3306

# Database setup
make migration-up      # Run database migrations
make migration-status  # Check migration status
make migration-new     # Create new migration
```

### Development Workflow
```bash
make test              # Run all tests (unit and integration)
make openapi           # Generate API code from OpenAPI spec
make start             # Start development server on port 8000
make demo              # Run error handling demo
```

### Running Individual Tests
```bash
# Domain tests (pure unit tests)
go test ./domain/...

# Application tests
go test ./app/...

# Integration tests
go test ./adapters/...

# Run specific test with verbose output
go test -v ./domain/task/...

# Run test with coverage
go test -cover ./...
```

## Architecture Overview

### Clean Architecture Layers

**Domain Layer (`domain/`)**
- Pure business logic with no external dependencies
- Contains `task` and `user` entities with business rules
- Defines repository interfaces and business operations
- Test with black-box approach, use table-driven tests

**Application Layer (`app/`)**
- CQRS pattern: Commands for write operations, Queries for read operations
- Commands: `AddUser`, `CreateTask`, `ChangeTaskStatus`, `AssignTask`
- Queries: `Tasks` for retrieving data
- Orchestrates domain logic without knowing infrastructure

**Ports (`ports/`)**
- HTTP server using Chi router with OpenAPI 3.0 specification
- Code generated by `oapi-codegen` from `ports/openapi.yml`
- Handles authentication middleware and Swagger documentation
- Entry point for external HTTP requests

**Adapters (`adapters/`)**
- Infrastructure implementations of domain interfaces
- MySQL repositories with sqlx driver
- HTTP client for external service calls
- Implements persistence and external integrations

**Core (`core/`)**
- Shared infrastructure: authentication, logging, error handling
- OpenTelemetry tracing with Jaeger backend
- JWT-based authentication system

### Key Patterns

1. **Dependency Inversion**: Inner layers define interfaces, outer layers implement
2. **CQRS**: Separate command and query handlers
3. **Black-box Testing**: Test only exported interfaces, use parallel test execution
4. **Code Generation**: Type-safe API handlers from OpenAPI spec

## Database Schema

The application uses MySQL with the following main tables:
- `users`: Stores employer and employee accounts with roles
- `tasks`: Task records with assignment and status tracking

Configuration in `migrations/dbconfig.yml` expects MySQL on `localhost:3306`.

## OpenAPI Specification

The API is defined in `ports/openapi.yml`. After modifications:
1. Run `make openapi` to regenerate type-safe handlers
2. Generated files: `openapi_types.gen.go` and `openapi_api.gen.go`

## Environment Configuration

Required environment variables (see `.env`):
- `MYSQL_ADDR`: Database host (default: localhost)
- `MYSQL_USER`: Database username (default: root)
- `MYSQL_PASSWORD`: Database password
- `MYSQL_DATABASE`: Database name
- `JWT_SECRET`: Secret for JWT token signing
- `JAEGER_ENDPOINT`: Jaeger tracing UI endpoint
- `PORT`: Server port (default: 8000)

## Testing Strategy

### Unit Tests
- Focus on domain layer business logic
- Use table-driven tests for comprehensive coverage
- Test only exported code (black-box testing)
- Package name suffix `_test` for test isolation

### Integration Tests
- Test adapter implementations with real MySQL
- Designed for parallel execution with `t.Parallel()`
- Avoid race conditions by not asserting on collection sizes
- Use specific ID lookups instead of length assertions

## Development Notes

- Always regenerate OpenAPI code after modifying `ports/openapi.yml`
- Database migrations should be idempotent and reversible
- New domain logic should include comprehensive unit tests
- Integration tests should support parallel execution
- Use structured logging with logrus throughout the codebase
- All HTTP handlers include OpenTelemetry tracing for observability
